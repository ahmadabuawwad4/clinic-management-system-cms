## Description

CMS is designed to replace paper-based and spreadsheet-driven clinic workflows with a single, role-based system. Administrators, receptionists, doctors, and accountants each get an interface scoped to their responsibilities, while the system enforces conflict-free appointment scheduling, accurate billing, and centralized patient/visit history. Medical records, prescriptions, lab, and radiology results are not authored in CMS — they are generated by an external report engine and linked back into the system against the relevant patient and visit.

## Features

- **Authentication & Access Control** — secure login, role-based access control (RBAC), account lockout after repeated failed attempts, login audit trail
- **User Management** — create/update/deactivate staff accounts, assign roles, secure password handling
- **Patient Management** — registration, search, duplicate detection, demographic and emergency contact records, full visit/appointment history
- **Doctor Management** — doctor profiles, specialization data, working-schedule configuration
- **Appointment Scheduling** — conflict-free booking, doctor availability checks, reschedule/cancel workflows
- **Visit Management** — appointment-based or walk-in visits, visit status tracking, administrative notes
- **Payment & Billing** — fee calculation, partial/full payment recording, balance validation, receipts
- **External Report Integration** — linked medical/prescription/lab/radiology reports tied to the correct patient and visit

## Installation

These steps assume Visual Studio with the ASP.NET and Node.js workloads installed.

1. **Clone the repository**
   ```bash
   git clone https://github.com/ahmadabuawwad4/clinic-management-system-cms.git
   cd clinic-management-system-cms
   ```

2. **Backend setup (.NET 8 / EF Core)**
   - Open the solution (`.sln`) file in Visual Studio.
   - Let Visual Studio restore NuGet packages automatically, or run:
     ```bash
     dotnet restore
     ```
   - Update the SQL Server connection string in `appsettings.json` (or `appsettings.Development.json`) to point to your local instance.
   - Apply EF Core migrations to create the database:
     ```bash
     dotnet ef database update
     ```
   - Run the backend by pressing **F5** in Visual Studio, or from the CLI:
     ```bash
     dotnet run
     ```

3. **Frontend setup (React)**
   ```bash
   cd client
   npm install
   npm start
   ```
   > Adjust the `client` folder path above if your React app lives in a different directory in the repo.

4. Confirm the API is running (check the port shown in the Visual Studio output/console) and that the React app can reach it — update the frontend's API base URL if needed.

## Usage

Example commands for day-to-day development:

```bash
# Run backend in watch mode
dotnet watch run

# Create a new EF Core migration after model changes
dotnet ef migrations add <MigrationName>

# Apply migrations to the database
dotnet ef database update

# Run the React frontend in development mode
npm start

# Build the React frontend for production
npm run build
```

## Configuration

Key settings to review before running locally or deploying:

- **Connection string** — SQL Server connection string in `appsettings.json`
- **JWT / auth settings** — token signing key and expiry, if applicable
- **CORS** — allowed origins for the React frontend to call the API
- **Frontend API base URL** — set in the React app's environment configuration (e.g., `.env`)
